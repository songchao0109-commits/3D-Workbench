# 产品变更记录

本文件用于记录 3D 影视分镜工作台的产品决策、范围变化和重要需求调整。

任何开发任务只要改变了产品行为、功能范围、交互体验、数据结构或验收标准，都应在这里增加记录。

## 2026-06-22

### 决策：产品方向

- 产品定位为面向影视流程的轻量 3D 分镜预演工作台。
- 不做完整 3D 建模软件。
- 核心价值是帮助用户在 AI 视频生成前确认空间、构图、机位和镜头运动。

### 决策：MVP 范围

- 导入格式：仅支持 `.glb`。
- 输出内容：参考帧和结构化摄影机/镜头参数。
- 编辑能力：支持模型变换和静态材质颜色调整。
- 时间线能力：需要关键帧时间线。
- AI 集成：第一版不直接接入 AI 视频生成接口。

### 决策：文档规则

- 本项目每次开发都必须形成或更新专业产品需求文档。
- 代码开发前应确认对应需求是否已写入 PRD。
- 涉及产品范围、交互、数据结构或验收标准变化时，必须更新本变更记录。

### 决策：项目文档语言

- 项目下的产品记录、开发记录和需求文档尽量使用中文。
- 如因工程生态需要保留英文文件名或代码标识，可以保留，但正文说明优先使用中文。

### 决策：方案确认与开发规则

- 沟通确认产品方案的过程中，每次方案变动都必须写入产品变更记录。
- 正式开发前必须与用户确认最终产品方案、技术架构、技术方案和数据存储结构。
- 未确认最终方案和技术路径前，不进入正式编码开发。
- 开发过程中必须保持模块解耦，拆分原子功能，避免把 UI、状态、3D 渲染、导入导出和数据存储混在一起。
- 技术实现需要为后续扩展预留空间，包括更多模型格式、更多关键帧通道、更多导出方式和未来 AI 视频接口。

### 变更：收紧 LookAt 交互边界并升级目标选择器

- 变更内容：相机注视目标选择器从“独立搜索框 + 列表”升级为 Suggest 一体化输入下拉组件，提示文案统一为“搜索对象”；同时在 LookAt 模式下收紧视口拖拽模式，明确旋转按钮禁用语义，并提示通过移动机位或调整注视目标改变朝向。
- 变更原因：用户指出当前按钮逻辑混乱，并要求优化注视目标资产列表形态，减少理解成本和无效操作。
- 影响范围：相机属性面板交互、LookAt 模式状态表达、目标搜索效率和相关验收标准。
- 当前结论：LookAt 模式下保留“移动”作为视口主编辑方式，旋转改为结果展示；目标选择统一通过 Suggest 组件完成，支持搜索对象与机位并单选回填。
- 状态：已完成。

### 变更：优化 FOV 编辑交互

- 变更内容：为相机 FOV 增加“滑杆 + 数值输入”双通道编辑，并将滑杆更新改为更实时的联动方式。
- 变更原因：真实交互走查发现单独依赖滑杆时，FOV 编辑反馈不稳定，不利于精细调整和可用性验证。
- 影响范围：相机属性面板、机位预览联动、FOV 验收路径。
- 当前结论：用户可以通过滑杆快速粗调，也可以通过数值框精调；FOV 有明确上下限并实时同步到相机状态。
- 状态：已完成。

### 变更：重构相机面板的信息层级与朝向交互

- 变更内容：将“全景图背景”上移到相机属性靠前位置；将原“旋转模式”重构为“镜头朝向”，选项调整为“自由朝向 / 注视目标”；在注视目标模式下不再暴露手动旋转拖拽入口，仅保留移动和目标调整。
- 变更原因：用户反馈当前“视口拖拽模式”和“旋转模式”叠加后存在理解成本，操作感觉像“先点 LookAt 再移动”，不符合常见 3D / 虚拟摄影工具的交互心智。
- 影响范围：相机属性面板结构、命名语义、拖拽交互和相关验收标准。
- 当前结论：参考 Unreal 与 Unity Cinemachine 的相机跟踪模式，`LookAt` 应被表达为朝向约束策略，而不是另一个手动旋转入口。
- 状态：已完成。

### 变更：确认快照工作区与右侧 Tabs 方案

### 变更：修正 FK 模式下被 IK 求解覆盖的问题

- 变更内容：骨骼运行时同步逻辑调整为仅在 `IK` 模式下执行 `IK` 骨链求解；`FK` 模式下不再自动运行已启用的 `IK` 链。
- 变更原因：真实交互测试发现，`FK` 模式旋转上级骨骼时，下级手臂 / 手指没有按层级跟随，表现像 `FK / IK` 混合。原因是每次同步骨骼状态后仍无条件执行 `IK` 求解，导致 `FK` 旋转结果被 `IK` 目标重新拉回。
- 影响范围：带骨骼模型的 `FK` 旋转、`IK` 模式切换、骨骼运行时同步规则。
- 当前结论：`FK` 与 `IK` 是互斥编辑模式；`FK` 模式只应用骨骼层级旋转，`IK` 模式才执行骨链求解。
- 状态：已确认，已修正。

### 变更：修正 FK / IK 模式下的控制对象绑定与拾取边界

- 变更内容：修正 `FK / IK` 模式切换后默认控制对象的绑定逻辑；同时将视口中的骨骼 / `IK` 控制点拾取从“单纯射线优先命中”调整为“射线命中 + 屏幕距离阈值”双重判断，避免控制点过度抢占模型本体的选择。
- 变更原因：真实交互测试发现，近期为了提升骨骼和 `IK` 点的可点击性，导致两类回归：一是切换到 `FK / IK` 模式后 gizmo 没有立即进入对应编辑态；二是点击模型时容易被骨骼或 `IK` 控制点误抢，造成整体对象和骨骼编辑边界混乱。
- 影响范围：骨骼控制主流程、视口拾取规则、对象整体编辑与骨骼子编辑的切换体验。
- 当前结论：`FK / IK` 模式切换后应默认进入对应控制态；仅当用户点击足够接近骨骼或 `IK` 控制点时，才进入骨骼子编辑，否则应保留对象整体选择能力。
- 状态：已确认，已修正。

### 变更：修正 IK 控制点与模型整体变换的实时同步

- 变更内容：当带骨骼模型在 `IK` 模式下进行整体移动、旋转或缩放时，系统在更新对象状态后会同帧刷新运行时骨骼与 `IK` 目标点，确保 `IK` 控制点始终保持与模型的当前相对关系，不再出现模型已移动但控制点停留在旧位置的现象。
- 变更原因：真实模型验收中发现，`IK` 控制点本身支持独立拖拽后，模型整体移动时没有同步保持当前相对坐标，导致视口中出现脱节的控制点。
- 影响范围：带骨骼 `.glb` 模型的整体变换交互、`IK` 编辑体验、视口运行时同步链路与自测验收标准。
- 当前结论：`IK` 目标点的屏幕表现必须与对象整体变换实时一致；后续涉及骨骼控制运行时同步的修改，均需以“模型整体操作优先保持控制点相对关系”为约束。
- 状态：已确认，已修正。

### 变更：修复带骨骼 GLB 整体动画插帧与关键帧编辑可见性

- 变更内容：调整手动插帧判断逻辑，带骨骼 GLB 在未进入骨骼控制态时默认记录对象整体 `position / rotation / scale`，只有明确选中骨骼或 IK 控制点时才记录 FK / IK 骨骼关键帧；同时将时间线上可见关键帧点改为均可选中、删除和拖动，聚合轨道操作只作用于该聚合点携带的引用。
- 变更原因：用户自测发现上传 GLB 后无法制作对象动画，且可见关键帧点不能移动和删除；排查确认旧逻辑对带骨骼对象过早进入骨骼插帧分支，且关键帧编辑入口隐藏在详细通道中，导致用户在汇总 / 对象轨上操作时像是功能未实现。
- 影响范围：手动插帧分支、带骨骼模型对象动画、时间线关键帧选中态、删除按钮启用逻辑、关键帧拖动交互和 M9 PRD 验收口径。
- 当前结论：带骨骼模型默认可作为普通对象制作整体动画；骨骼动画只在骨骼控制态触发；时间线中用户能看到的关键帧点都应能直接编辑。
