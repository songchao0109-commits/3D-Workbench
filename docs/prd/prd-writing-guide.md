# 需求评审级 PRD 写作指南

## 1. 写作定位

本项目的产品文档不应只是功能说明列表，而应是可以用于需求评审的协作文件。

一份合格的需求评审级 PRD 要回答四类问题：

1. 为什么要做：业务背景、用户问题、证据、机会判断。
2. 做到什么程度：目标、成功指标、范围、非范围、优先级。
3. 怎么做才算对：用户流程、交互方案、功能规则、数据结构、验收标准。
4. 怎么协作落地：技术约束、依赖、风险、排期、评审结论、变更记录。

如果文档只能告诉工程“有哪些按钮和字段”，但不能帮助产品、设计、工程、测试判断“这个需求是否值得做、是否想清楚、是否可验收”，则不满足本项目的 PRD 标准。

## 2. 参考来源

本规范参考了以下公开资料，并结合本项目“3D 影视分镜工作台”的开发协作方式做了裁剪：

- ProcessOn 的需求文档说明强调：需求文档应包含文档说明、修订历史、项目背景、用户角色、业务流程、功能需求、非功能需求、原型与交互、数据需求、约束与风险等内容。
  来源：https://www.processon.com/knowledge/xuqiudocument
- Reddit 产品经理讨论中的共识是：优秀 PRD 不只是模板，而是清楚表达问题、背景、目标、取舍、用户价值和验证方式；过长但没有判断的信息并不能提高质量。
  来源：https://www.reddit.com/r/ProductManagement/comments/1g8v0oc/what_are_some_excellent_examples_of_good_prds/?tl=zh-hans
- Atlassian Confluence 的 PRD 模板强调：PRD 应定义产品或功能的目的、功能和范围，并包含假设、用户故事、UX 设计、开放问题和范围控制。
  来源：https://www.atlassian.com/software/confluence/templates/product-requirements
- Aha! 的 PRD 指南强调：现代 PRD 应保持轻量和可迭代，通常包含概览、目标、上下文、假设、范围、需求、成功指标和开放问题。
  来源：https://www.aha.io/roadmapping/guide/requirements-management/what-is-a-good-product-requirements-document-template
- ProductPlan 的 PRD 说明强调：PRD 应说明每个版本必须包含的能力，每个能力最好配套用例，并补充系统环境要求、可用性要求、假设、约束和依赖。
  来源：https://www.productplan.com/glossary/product-requirements-document

## 3. 文档分层与使用边界

本项目采用“产品基线 + 阶段确认 + 变更记录”的产品文档协作方式。写文档时先判断文档类型，再选择结构，不要求所有内容都套同一个大模板。

### 3.1 产品总 PRD

文件：`docs/prd/3d-workbench-prd.md`

用途：

- 定义产品定位、目标用户、核心问题、长期范围和总体能力地图。
- 作为所有阶段开发的产品基线。
- 不应频繁写成开发日报，也不应只记录某次小改动。

适用内容：

- 产品定位、目标用户和核心用户问题。
- MVP 范围、长期非目标和能力地图。
- 关键产品决策。
- 总体信息架构、数据模型草案和里程碑。

### 3.2 阶段/功能 PRD

文件示例：

- `docs/prd/m1-development-confirmation.md`
- `docs/prd/m2-development-confirmation.md`
- `docs/prd/m3-camera-editing-confirmation.md`
- `docs/prd/m4-skeleton-control-confirmation.md`

用途：

- 针对某一开发阶段或较大功能做需求评审。
- 必须包含背景、目标、范围、用户流程、产品方案、功能规则、数据结构、技术约束、验收标准和风险。
- 开发前必须从“待确认”变成“已确认”。

适用内容：

- 一个阶段的开发前确认。
- 一个较大功能的完整方案。
- 会影响数据结构、技术架构、用户流程或验收标准的需求变化。

### 3.3 轻量开发确认

当需求较小、影响范围明确、不会改变核心数据结构或技术架构时，可以使用轻量开发确认结构，不必套完整阶段/功能 PRD。

建议结构：

1. 文档信息。
2. 背景与问题。
3. 本次要做与不做。
4. UI 或交互方案。
5. 技术影响。
6. 验收标准。
7. 风险与待确认问题。

适用内容：

- 已有能力的小范围增强。
- UI 布局或交互微调。
- 已确认方案下的补充开发。
- 不改变产品方向的小修小补。

### 3.4 变更日志

文件：`docs/prd/change-log.md`

用途：

- 记录沟通过程中每一次产品方案变化。
- 记录变更原因、影响范围和当前结论。
- 防止口头决策丢失。

## 4. 需求评审级 PRD 必备结构

每份阶段/功能 PRD 建议使用 `docs/prd/prd-template.md`，核心结构如下：

1. 文档控制
   - 文档版本、状态、负责人、创建/更新日期。
   - 评审状态：草案 / 待评审 / 已确认 / 开发中 / 已完成 / 已废弃。
   - 相关文档和变更记录。

2. 一页摘要
   - 一句话说明本次要解决什么问题。
   - 本次为什么值得做。
   - 本次交付什么，不交付什么。
   - 当前最重要的风险或未决问题。

3. 背景、问题与依据
   - 业务背景。
   - 用户问题。
   - 当前方案或竞品方案的不足。
   - 证据来源：用户反馈、截图、竞品、调研、数据、内部判断。

4. 目标用户、场景与用户旅程
   - 主要用户和次要用户。
   - 用户角色、目标、痛点。
   - 典型使用场景。
   - 触发条件和使用频率。
   - 用户从触发到完成任务的关键步骤。

5. 目标、非目标与成功指标
   - 产品目标：用户能完成什么任务。
   - 体验目标：操作路径、清晰度、效率、稳定性。
   - 非目标：本次明确不追求什么。
   - 成功指标：可量化则量化，不可量化则定义可观察结果。

6. 范围、优先级与版本边界
   - P0：没有就不能交付。
   - P1：重要但可以分批。
   - P2：增强项或后续优化。
   - 明确本次不做内容，避免评审时范围失控。
   - 明确哪些内容进入后续版本。

7. 产品方案与用户流程
   - 产品方案：页面结构、核心交互、状态变化。
   - 设计方案：信息架构、布局、视觉参考、关键控件。
   - 用户从哪里进入、如何完成主路径。
   - 分支路径、异常路径和状态说明。
   - 可用流程图、状态图或步骤表。

8. 功能需求与规则
   - 按用户任务或模块拆分。
   - 每个功能说明入口、流程、规则、边界、异常和权限。
   - 避免只写“支持某功能”，必须写清楚用户如何使用、系统如何响应。

9. 数据、技术与非功能要求
    - 数据模型、字段说明、存储方式。
    - 导入导出格式、数据迁移或兼容策略。
    - 技术架构、模块边界、关键依赖、实现策略。
    - 性能：加载、渲染、响应、文件大小约束。
    - 兼容性：浏览器、设备、文件格式。
    - 可用性：错误提示、空状态、禁用状态。
    - 可维护性：模块解耦、扩展策略。
    - 安全与隐私：本地文件、导出数据、未来接口。

10. 验收、风险、开放问题与评审记录
    - 使用可验证的“给定/当/则”写法。
    - 覆盖主路径、异常路径、边界条件和数据输出。
    - 验收项必须能被人工或自动测试验证。
    - 任务拆分。
    - 阶段交付物。
    - 评审节点。
    - 验收方式。
    - 风险：可能失败或返工的点。
    - 假设：当前暂时当作成立的判断。
    - 依赖：素材、技术库、接口、浏览器能力、用户确认。
    - 开放问题：没有确认前不得进入开发范围。
    - 评审结论。
    - 参与方。
    - 变更项。
    - 决策原因。

## 5. 需求评审门禁

进入正式开发前，阶段/功能 PRD 必须通过以下门禁：

### 5.1 产品门禁

- 问题是否真实、具体、值得本阶段解决。
- 目标用户和使用场景是否清楚。
- 本次范围和非范围是否明确。
- 是否有足够依据支撑方案，而不是只凭直觉堆功能。

### 5.2 设计门禁

- 主流程是否闭环。
- 信息架构是否清晰。
- 空状态、错误状态、加载状态、禁用状态是否被考虑。
- 视觉参考是否被归档，不能只存在聊天记录里。

### 5.3 技术门禁

- 技术架构是否明确。
- 模块边界是否清楚。
- 数据结构是否可支撑后续扩展。
- 性能、兼容性、依赖风险是否被记录。

### 5.4 验收门禁

- 是否有可验证的验收标准。
- 是否覆盖主路径和异常路径。
- 是否说明测试方式或人工验证方式。
- 是否记录了未覆盖风险。

未通过门禁时，不进入正式编码开发。

## 5.5 组件规范

- 本项目下的界面组件优先使用 `elementdesign` 组件库。
- 当项目尚未接入 `elementdesign`，或当前场景缺少可用组件时，允许先用自定义轻量实现，但应保持接口、层级和交互尽量可替换，避免后续迁移成本过高。
- 在评审文档和开发实现中，如果没有使用 `elementdesign`，应能说明原因，例如：当前仓库未接入、目标组件缺失、3D 视口场景需要定制交互等。

## 6. 功能需求写法

功能需求不能只写“支持某能力”，而要写到开发可以据此实现逻辑、测试可以据此验收的粒度。

每个功能模块建议先使用基础格式，再按组件类型补充专项规格。基础格式回答“这个功能为什么存在、怎么用、有什么规则”；专项规格回答“这个功能里的控件、状态、数据、联动和异常到底怎么实现”。

```markdown
### 功能名称

用户问题：

- 用户为什么需要这个功能。

用户故事：

- 作为【用户类型】，我希望【完成动作】，以便【获得价值】。

入口：

- 用户从哪里进入这个功能。

主流程：

1. 第一步。
2. 第二步。
3. 第三步。

规则：

- 业务规则。
- 状态规则。
- 数据规则。

规格明细：

| 维度 | 说明 |
| --- | --- |
| 展示内容 | 页面、区域、字段、控件、默认值、占位文案 |
| 数据来源 | 接口、本地状态、文件、运行时对象、用户输入、派生计算 |
| 数据规则 | 字段口径、格式化、默认值、缺失值、去重、排序、过滤、缓存 |
| 交互规则 | 点击、拖拽、输入、选择、快捷操作、撤销、确认、取消 |
| 状态规则 | 默认、选中、悬停、禁用、加载、成功、失败、空状态 |
| 权限规则 | 可见、可操作、只读、锁定、无权限提示 |
| 联动规则 | 与其他模块、列表、详情、视口、时间线、导出结果的同步关系 |
| 持久化规则 | 是否保存、保存位置、刷新后是否保留、导入导出是否包含 |
| 性能约束 | 数据量、文件大小、刷新频率、响应时间、渲染成本 |

边界与异常：

- 空状态。
- 错误状态。
- 禁用状态。
- 文件或数据异常。

验收标准：

- 给定【前置条件】，当【用户操作】，则【可观察结果】。
```

### 6.1 组件专项规格

一个功能模块可能包含多个组件类型。写 PRD 时应按实际命中的类型选择补充，不需要每次全部填写。

#### 数据列表 / 表格 / 卡片流

必须说明列表目的、数据来源、首次加载时机、字段展示规则、默认排序、可切换排序、分页方式、每页数量、搜索筛选、空状态、行操作、批量操作、刷新策略，以及新增、编辑、删除后的回写策略。

典型问题：

- 空状态是无数据、无搜索结果、无权限，还是加载失败。
- 排序发生在前端还是后端，默认字段是什么。
- 分页是页码、游标、无限滚动，切页后是否保留筛选条件。

#### 详情页 / 详情面板

必须说明入口、数据来源、字段分组、字段展示规则、编辑入口、只读与锁定状态、缺失数据展示、与列表或视口的联动，以及关闭、返回或切换对象后的状态保留。

#### 表单 / 属性编辑器

必须说明字段列表、字段类型、默认值、必填规则、校验规则、输入限制、提交时机、保存成功反馈、保存失败处理、重置、取消、撤销和脏数据提醒。

典型问题：

- 修改是实时生效、失焦保存，还是点击保存后生效。
- 校验失败时阻断提交，还是允许保存草稿。
- 取消后恢复到进入页面前，还是恢复到上一次保存状态。

#### 搜索 / 筛选 / 排序

必须说明搜索范围、触发方式、防抖时间、筛选项来源、多筛选条件关系、默认筛选、清空规则、与分页的联动，以及是否同步到 URL、本地状态或项目状态。

#### 上传 / 导入 / 文件解析

必须说明支持格式、文件大小限制、数量限制、选择入口、是否支持拖拽、解析流程、进度展示、成功后的数据写入、失败原因与提示、重试、取消、清理和临时资源释放策略。

#### 导出 / 下载 / 生成结果

必须说明导出入口、导出对象、文件格式、文件命名、字段内容、数据来源、导出前校验、生成中状态、成功反馈、失败处理，以及导出内容与当前视图、选中项或筛选条件的关系。

#### 弹窗 / 抽屉 / 浮层

必须说明打开入口、打开条件、内容结构、默认聚焦、确认、取消、关闭规则、点击遮罩或按 Esc 的行为、未保存内容处理、多层弹窗限制，以及移动端或小屏适配。

#### 导航 / 标签页 / 步骤流

必须说明入口与层级、默认选中项、切换规则、是否允许跳步、未完成步骤限制、返回规则、状态保留，以及是否同步 URL 或项目状态。

#### 选择 / 批量操作

必须说明可选择对象、单选或多选、默认选中、选择上限、全选范围、批量操作项、执行前确认、部分成功与部分失败处理，以及操作后的选中状态。

#### 状态机 / 工作流

必须说明状态列表、初始状态、状态流转条件、可逆与不可逆操作、每个状态下的可见内容、每个状态下的可用操作、异常状态和状态变更记录。

#### 画布 / 3D 视口 / 拖拽交互

必须说明可交互对象、选择规则、拖拽、旋转、缩放规则、坐标系与单位、吸附、约束、边界、辅助器显示、选中、高亮、锁定、隐藏状态，以及与属性面板、对象列表、时间线的同步。

#### 时间线 / 播放 / 关键帧

必须说明时间单位、默认时长、FPS、播放头规则、可关键帧对象、可关键帧属性、添加、更新、删除关键帧规则、插值方式、播放、暂停、跳转，以及与视口和属性面板的同步。

#### 权限 / 可见性 / 锁定

必须说明权限来源、角色或状态差异、可见规则、可操作规则、只读规则、锁定规则、无权限提示，以及导出、保存、删除等高风险操作限制。

#### 通知 / 提示 / 反馈

必须说明触发场景、提示类型、成功文案、失败文案、是否自动消失、是否提供操作入口，以及多条提示合并策略。

#### 设置 / 配置 / 偏好

必须说明配置项、默认值、可选值、生效时机、作用范围、是否持久化、重置规则，以及与已有数据的兼容策略。

## 7. 验收标准写法

验收标准要能被手动或自动验证。推荐使用“给定/当/则”的句式：

```markdown
- 给定用户已选择合法 `.glb` 文件，当导入完成，则模型应出现在 3D 视口和对象列表中。
- 给定用户选中模型，当修改材质颜色，则视口中的模型颜色应实时更新，并在项目状态中记录材质覆盖。
- 给定用户导出镜头参数，当导出完成，则 JSON 中应包含 `schemaVersion`、`shots`、`cameras` 和 `keyframes`。
```

避免以下写法：

- “交互要丝滑。”
- “导入体验好。”
- “时间线功能完善。”
- “支持模型编辑。”

这类描述可以作为体验目标，但不能直接作为验收标准。

## 8. 变更记录写法

每次方案变化都要写入 `docs/prd/change-log.md`。建议格式：

```markdown
### 变更：一句话说明

- 变更内容：
- 变更原因：
- 影响范围：
- 当前结论：
- 状态：已确认 / 待确认 / 已废弃
```

如果只是想法，还没有确认，应标记为“待确认”，不得进入开发范围。

## 9. 本项目特别要求

由于本项目涉及 3D 渲染、时间线、导入导出和未来 AI 视频接口，PRD 必须额外说明：

- 3D 场景对象的数据结构。
- Three.js 运行时对象与产品状态的映射关系。
- 摄影机数据结构。
- 关键帧数据结构。
- 材质覆盖数据结构。
- 项目 JSON 和镜头 JSON 的导出格式。
- 渲染层、状态层、UI 层、导入导出层的模块边界。
- 当前版本不支持的 3D 能力，例如骨骼动画、网格编辑、复杂材质节点。
- 性能边界，例如 GLB 文件大小、模型面数、材质数量、浏览器兼容性。

## 10. PRD 质量检查清单

开发前检查：

- 是否有一页摘要，能让评审者快速理解本次需求。
- 是否说明了为什么做。
- 是否说明了为谁做。
- 是否有证据或依据，而不是只有主观判断。
- 是否明确本次做什么和不做什么。
- 是否有用户故事或关键用例。
- 是否有用户流程或业务流程。
- 是否每个功能都有验收标准。
- 是否每个功能都写清展示内容、数据来源、交互规则、状态规则、联动规则和持久化规则。
- 是否按组件类型补充了专项规格，例如列表、表单、上传、导出、弹窗、批量操作、状态机、3D 视口或时间线。
- 是否说明非功能需求。
- 是否说明数据结构和存储方式。
- 是否说明技术约束、依赖和风险。
- 是否记录开放问题。
- 是否更新变更日志。
- 是否已经与用户确认最终产品方案和技术方案。

如果以上任一关键项缺失，应先补齐文档，再进入开发。
