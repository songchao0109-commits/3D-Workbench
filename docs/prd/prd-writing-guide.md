# PRD 写作指南

## 1. 写作定位

本项目的产品文档用于说明需求本身，方便讨论、开发实现和测试验收。文档不记录讨论流程，也不在正文中放置评审状态、评审意见、相关文档列表或相关变更记录。

一份合格的 PRD 要回答四类问题：

1. 为什么要做：背景、用户问题、依据。
2. 做什么和不做什么：一页摘要、交付内容、不交付内容、功能优先级。
3. 怎么做才算对：用户流程、功能规则、状态规则、异常规则、联动规则。
4. 怎么验证：验收矩阵、回归检查、测试注意事项。

如果文档只能告诉工程“有哪些按钮和字段”，但不能说明“不同状态下如何展示、数据从哪里来、操作后如何联动、异常时如何处理、测试如何验证”，则不满足本项目的 PRD 标准。

## 2. 文档分层与使用边界

本项目采用“产品基线 + 阶段/功能 PRD + 变更记录”的文档协作方式。正式 PRD 只写需求内容，变更过程放在独立变更记录里。

### 2.1 产品总 PRD

文件：`docs/prd/3d-workbench-prd.md`

用途：

- 定义产品定位、目标用户、核心问题、长期能力地图。
- 作为阶段/功能 PRD 的产品基线。
- 不写成开发日报，也不记录每次小修改。

### 2.2 阶段/功能 PRD

用途：

- 针对某一开发阶段或较大功能说明需求。
- 必须包含文档信息、一页摘要、背景依据、用户场景、产品方案、功能规则、非功能性需求和测试验收。
- 不在文档中写相关文档、相关变更记录、评审状态或评审意见。

### 2.3 轻量开发确认

当需求较小、影响范围明确、不会改变核心流程时，可以使用轻量结构：

1. 文档信息。
2. 一页摘要。
3. 背景与问题。
4. 本次交付与不交付。
5. 功能规则。
6. 非功能性需求。
7. 测试验收。

## 3. 标准结构

阶段/功能 PRD 使用 `docs/prd/prd-template.md`，核心结构如下：

1. 文档信息
   - 用表格记录版本修订历史。
   - 表头固定为：文档版本、修订日期（YYYY/MM/DD）、变更内容、变更原因、备注。

2. 一页摘要
   - 一句话结论。
   - 本次解决的问题。
   - 功能优先级。
   - 本次交付内容。
   - 本次不交付内容。
   - 关键风险或待确认问题。

3. 背景、问题与依据
   - 背景。
   - 用户问题。
   - 现有方案不足。
   - 证据与依据。

4. 用户、场景与用户旅程
   - 用户角色。
   - 使用场景。
   - 触发条件。
   - 用户旅程。

5. 产品方案与用户流程
   - 产品方案。
   - 页面/区域结构。
   - 主流程。
   - 分支流程。
   - 异常流程。
   - 状态说明。
   - 流程图。

6. 功能需求与规则
   - 功能需求索引。
   - 每个功能的用户问题、用户故事、入口、流程、规则、状态机、规格明细、边界与异常。
   - 组件类型专项清单。

7. 非功能性需求
   - 性能要求。
   - 兼容性要求。
   - 可用性要求。
   - 可维护性要求。
   - 安全与隐私要求。
   - 可观测性要求。

8. 测试验收
   - 验收矩阵。
   - 回归检查。
   - 测试注意事项。

## 4. 写作禁区

正式 PRD 中不要写以下内容：

- 相关文档列表。
- 相关变更记录列表。
- 文档状态、评审状态、当前评审结论。
- 评审参与方、评审意见、评审记录。
- 独立的目标、非目标与成功指标章节。
- 独立的范围、优先级与版本边界章节。
- 数据契约、字段说明、数据流与副作用等技术规格大章。
- 决策记录。
- 文档末尾的变更记录。

这些信息可以在沟通、变更日志或开发文档中维护，但不要进入正式 PRD 主体。

## 5. 功能需求写法

功能需求不能只写“支持某能力”，而要写到开发可以实现逻辑、测试可以据此验收的粒度。

每个功能模块建议使用以下结构：

```markdown
### FR-001 功能名称

用户问题：

- 用户为什么需要这个功能。

用户故事：

- 作为【用户类型】，我希望【完成动作】，以便【获得价值】。

入口：

- 用户从哪里进入这个功能。

主流程：

1. 第一步。
2. 第二步。
3. 第三步。

规则：

- 业务规则。
- 状态规则。
- 数据规则。
- 权限规则。
- 联动规则。

状态机：

| 状态 | 进入条件 | 可见内容 | 可用操作 | 退出条件 | 异常处理 |
| --- | --- | --- | --- | --- | --- |
| 默认 |  |  |  |  |  |

规格明细：

| 维度 | 说明 |
| --- | --- |
| 展示内容 | 页面、区域、字段、控件、默认值、占位文案、格式化方式 |
| 数据来源 | 接口、本地状态、文件、运行时对象、用户输入、派生计算、缓存 |
| 数据规则 | 字段口径、默认值、缺失值、去重、排序、过滤、冲突处理、精度 |
| 交互规则 | 点击、拖拽、输入、选择、快捷操作、撤销、确认、取消、禁用条件 |
| 状态规则 | 默认、选中、悬停、聚焦、禁用、加载、成功、失败、空状态 |
| 权限规则 | 可见、可操作、只读、锁定、无权限提示、高风险操作保护 |
| 联动规则 | 与其他模块、列表、详情、视口、时间线、导出结果的同步关系 |
| 持久化规则 | 是否保存、保存位置、保存时机、刷新后是否保留、导入导出是否包含 |
| 性能约束 | 数据量、文件大小、刷新频率、响应时间、渲染成本、降级策略 |

边界与异常：

- 空状态。
- 错误状态。
- 禁用状态。
- 文件或数据异常。
- 重复操作。
- 快速切换。
- 撤销或回退。
- 资源清理。
```

功能需求写完后，应能回答四个问题：

- 开发是否知道需要读写哪些状态和字段。
- 设计是否知道每种状态下页面展示什么。
- 测试是否能据此写出主路径、异常路径和边界用例。
- 产品是否能判断这条需求是否仍在本期交付内容内。

## 6. 组件专项规格

一个功能模块可能包含多个组件类型。写 PRD 时应按实际命中的类型选择补充，不需要每次全部填写。

### 数据列表 / 表格 / 卡片流

必须说明列表目的、数据来源、首次加载时机、字段展示规则、默认排序、可切换排序、分页方式、每页数量、搜索筛选、空状态、行操作、批量操作、刷新策略，以及新增、编辑、删除后的回写策略。

典型问题：

- 空状态是无数据、无搜索结果、无权限，还是加载失败。
- 排序发生在前端还是后端，默认字段是什么。
- 分页是页码、游标、无限滚动，切页后是否保留筛选条件。
- 新增、编辑、删除、批量操作后列表如何刷新，选中项是否保留。
- 大数据量下是否有虚拟滚动、加载上限或降级策略。

### 详情页 / 详情面板

必须说明入口、数据来源、字段分组、字段展示规则、编辑入口、只读与锁定状态、缺失数据展示、与列表或视口的联动，以及关闭、返回或切换对象后的状态保留。

### 表单 / 属性编辑器

必须说明字段列表、字段类型、默认值、必填规则、校验规则、输入限制、提交时机、保存成功反馈、保存失败处理、重置、取消、撤销和脏数据提醒。

### 搜索 / 筛选 / 排序

必须说明搜索范围、触发方式、防抖时间、筛选项来源、多筛选条件关系、默认筛选、清空规则、与分页的联动，以及是否同步到 URL、本地状态或项目状态。

### 上传 / 导入 / 文件解析

必须说明支持格式、文件大小限制、数量限制、选择入口、是否支持拖拽、解析流程、进度展示、成功后的数据写入、失败原因与提示、重试、取消、清理和临时资源释放策略。

### 导出 / 下载 / 生成结果

必须说明导出入口、导出对象、文件格式、文件命名、字段内容、数据来源、导出前校验、生成中状态、成功反馈、失败处理，以及导出内容与当前视图、选中项或筛选条件的关系。

### 弹窗 / 抽屉 / 浮层

必须说明打开入口、打开条件、内容结构、默认聚焦、确认、取消、关闭规则、点击遮罩或按 Esc 的行为、未保存内容处理、多层弹窗限制，以及移动端或小屏适配。

### 导航 / 标签页 / 步骤流

必须说明入口与层级、默认选中项、切换规则、是否允许跳步、未完成步骤限制、返回规则、状态保留，以及是否同步 URL 或项目状态。

### 选择 / 批量操作

必须说明可选择对象、单选或多选、默认选中、选择上限、全选范围、批量操作项、执行前确认、部分成功与部分失败处理，以及操作后的选中状态。

### 状态机 / 工作流

必须说明状态列表、初始状态、状态流转条件、可逆与不可逆操作、每个状态下的可见内容、每个状态下的可用操作、异常状态和状态变更记录。

### 画布 / 3D 视口 / 拖拽交互

必须说明可交互对象、选择规则、拖拽、旋转、缩放规则、坐标系与单位、吸附、约束、边界、辅助器显示、选中、高亮、锁定、隐藏状态，以及与属性面板、对象列表、时间线的同步。

### 时间线 / 播放 / 关键帧

必须说明时间单位、默认时长、FPS、播放头规则、可关键帧对象、可关键帧属性、添加、更新、删除、拖动关键帧规则、插值方式、轨道层级、片段或标记规则、播放、暂停、跳转、播放结束行为，以及与视口和属性面板的同步。

### 权限 / 可见性 / 锁定

必须说明权限来源、角色或状态差异、可见规则、可操作规则、只读规则、锁定规则、无权限提示，以及导出、保存、删除等高风险操作限制。

### 通知 / 提示 / 反馈

必须说明触发场景、提示类型、成功文案、失败文案、是否自动消失、是否提供操作入口，以及多条提示合并策略。

### 设置 / 配置 / 偏好

必须说明配置项、默认值、可选值、生效时机、作用范围、是否持久化、重置规则，以及与已有数据的兼容策略。

### AI / 自动生成 / 智能辅助

必须说明输入信号、用户可控参数、生成策略、输出形式、失败兜底、结果是否可编辑、是否需要解释信息、耗时和成本提示。

## 7. 验收标准写法

验收标准要能被手动或自动验证。推荐使用“给定/当/则”的句式：

```markdown
- 给定用户已选择合法 `.glb` 文件，当导入完成，则模型应出现在 3D 视口和对象列表中。
- 给定用户选中模型，当修改材质颜色，则视口中的模型颜色应实时更新，并在项目状态中记录材质覆盖。
- 给定用户点击导出动画，当当前版本尚未接入视频渲染，则系统应展示待接入提示且不生成文件。
```

避免以下写法：

- “交互要丝滑。”
- “导入体验好。”
- “时间线功能完善。”
- “支持模型编辑。”

这类描述可以作为体验方向，但不能直接作为验收标准。

## 8. 质量检查清单

开发前检查：

- 是否有一页摘要。
- 是否说明了为什么做。
- 是否说明了为谁做。
- 是否有证据或依据，而不是只有主观判断。
- 是否把功能优先级放在一页摘要中。
- 是否明确本次交付什么和不交付什么。
- 是否有用户故事或关键用例。
- 是否有用户流程或业务流程。
- 是否每个 P0 / P1 功能都有需求编号和测试项。
- 是否每个功能都写清展示内容、数据来源、交互规则、状态规则、联动规则和持久化规则。
- 是否写清复杂交互的状态机、异常路径和回退路径。
- 是否按组件类型补充了专项规格。
- 是否说明非功能性需求。
- 是否只在最后保留测试相关内容。
- 是否移除了相关文档、相关变更记录、评审状态、评审意见、数据契约和文档末尾变更记录。
