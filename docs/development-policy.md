# 开发约定

## 需求评审文档优先

本项目每一次开发都必须和产品文档绑定。

沟通确认阶段：

1. 需求沟通、方案讨论和范围确认过程中，只要方案发生变化，都必须记录到 `docs/prd/change-log.md`。
2. 变更记录应说明变更内容、原因、影响范围和当前结论。
3. 未确认的想法可以记录为“待确认”，不得直接进入开发范围。
4. 需求文档必须能支撑评审，而不是只罗列功能点。
5. 沟通过程中的截图、参考链接、竞品观察和关键判断，应归档到 PRD 或相关资料目录中。

开发前：

1. 确认本次工作是否已经被 `docs/prd/3d-workbench-prd.md` 覆盖。
2. 如果本次工作会改变功能范围、产品行为、交互体验、数据结构或验收标准，必须先更新 PRD。
3. 如果产生新的产品决策或范围变化，必须在 `docs/prd/change-log.md` 增加日期记录。
4. 开发前必须与用户确认最终产品方案。
5. 开发前必须与用户确认技术架构、技术方案和数据存储结构。
6. 开发前必须通过需求评审门禁，确认背景、目标、用户流程、范围、非范围、验收标准、风险和开放问题。
7. 未完成上述确认前，不进入正式编码开发。

开发中：

1. 代码实现必须和 PRD 中的验收标准保持一致。
2. 不主动增加未被需求确认的产品范围。
3. 对高影响的未决问题，应记录下来，不要在代码里默默做不可逆决策。
4. 功能实现应保持解耦，避免把视图、业务状态、数据存储、导入导出和 3D 渲染逻辑混在一起。
5. 开发时应拆分原子功能，优先完成可独立验证、可独立替换的小模块。
6. 关键模块应为后续扩展预留空间，例如更多模型格式、更多关键帧通道、更多导出格式和未来 AI 视频接口。

完成前：

1. 按 PRD 验收标准检查实现结果。
2. 记录未完成的验收项或已知限制。
3. 保持导出数据结构可版本化、可阅读、可追踪。

## 文档语言约定

项目下的产品记录、需求文档、开发记录和说明文档尽量使用中文。

允许保留英文的情况：

- 文件名、目录名、代码标识、接口字段、库名、命令名。
- 行业通用术语，例如 GLB、JSON、MVP、PRD、FOV。
- 为方便工程引用而保留的英文路径。

## 需求评审级产品文档标准

专业产品需求文档应尽量包含：

- 一页摘要。
- 业务背景与用户问题。
- 证据与依据。
- 目标用户、使用场景和用户旅程。
- 产品目标、体验目标和成功指标。
- 范围、非范围和优先级。
- 用户流程、业务流程和异常流程。
- 产品方案、设计方案、技术方案和数据方案。
- 功能需求和非功能需求。
- 可验证的验收标准。
- 排期、里程碑和交付物。
- 风险、假设、依赖和开放问题。
- 评审记录和变更记录。

详细写法参考：

- `docs/prd/prd-writing-guide.md`
- `docs/prd/prd-template.md`

## 技术方案确认标准

进入开发前，至少需要形成并确认以下内容：

- 最终产品方案：本次要做什么、不做什么、验收标准是什么。
- 技术架构：前端框架、3D 渲染方案、状态管理方式、模块边界。
- 技术方案：关键功能如何实现、依赖哪些库、有哪些限制和风险。
- 数据存储结构：项目 JSON、镜头 JSON、模型资产、材质覆盖、关键帧数据如何组织。
- 扩展策略：后续如何增加模型格式、时间线能力、导出能力和 AI 接口。

## 需求评审门禁

阶段/功能 PRD 进入开发前必须检查：

- 产品门禁：问题、目标用户、场景、范围、非范围是否清楚。
- 设计门禁：主流程、异常流程、信息架构、状态设计和视觉参考是否清楚。
- 技术门禁：架构、模块边界、数据结构、依赖、性能和兼容性风险是否清楚。
- 验收门禁：验收标准是否可验证，是否覆盖主路径、异常路径和关键数据输出。

未通过门禁时，只能继续补充文档和方案讨论，不能进入正式编码。

## 解耦与原子功能原则

开发过程中应遵守以下原则：

- 渲染层只负责 3D 场景显示和交互，不直接承担产品业务决策。
- 状态层负责项目、镜头、对象、摄影机、关键帧等核心数据。
- 导入导出层独立处理文件解析、图片导出和 JSON 序列化。
- UI 组件尽量围绕单一职责设计，例如对象列表、属性面板、时间线、导出面板分开实现。
- 每个功能尽量能独立验证，避免一次性提交过大的混合改动。
- 新增能力优先通过扩展已有模块接口实现，而不是改动大量既有逻辑。
